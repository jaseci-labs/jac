- **Feature: Solid (signals) client backend**: `framework = "solid"` now selects `SolidBackend`, the first fine-grained reactivity target for client codegen (`.cl.jac` / `cl {}` → JavaScript). Unlike the React/Preact re-render model, Solid runs each component body once and wires a reactive graph, so the backend exercises the read seams added in the prior phases: reactive state declarations lower to `createSignal`, value reads become accessor calls (`count()`) via `lower_state_read`, and dynamic JSX children/attributes become thunks (`() => count()`) via `lower_view_expr` so the runtime can re-track each binding. Writes route through the setter (`setCount(...)`); augmented assigns use a functional updater (`count += 1` → `setCount(prev => prev + 1)`) to avoid stale reads and self-subscription. Effects lower to `createEffect` (entry) and `onCleanup` (exit); the shared `__jacJsx` factory targets Solid's hyperscript runtime (`solid-js/h`), and `try/awaiting/except` slots lower to `<Suspense>`/`<ErrorBoundary>`. React/Preact output is unchanged (every new lowering is a no-op on the hooks backends). v1 is a best-effort subset over the representative fixtures; documented gaps (per the reactivity contract) include deferred `Ref[T]` fields (lowered to a plain `let` binding, no `ref={...}` wiring yet), block-level shadowing of reactive fields, the functional-updater asymmetry for plain self-referential assigns, true `createResource`-backed suspension, and full `dom-expressions` parity.
