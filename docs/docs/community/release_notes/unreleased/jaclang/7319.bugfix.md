- **Fix: `sorted`/`.sort` rejected inline lambda keys over `list[dict]`**: `key=lambda d: dict : d["n"]` (and the equivalent `.sort()` form) failed type-checking with an opaque `E1054: No matching overload found`, even though a named key function with the identical signature worked. Bare generics like `dict` already default their unresolved type parameters to `any`, but that `any` was not trusted by strict-mode assignability checks, so it was rejected against `sorted`'s `SupportsRichComparison` overload. The defaulted `any` is now trusted, fixing this and any other strict-mode call site affected by the same bare-generic defaulting, not just `sorted`/`.sort`.
- **Fix: `E1054` now explains genuinely unresolved lambda return types**: When a lambda argument passed to an overloaded call (such as `sorted`/`.sort`) has fully annotated parameters but its body's return type still cannot be resolved, `E1054` now carries a specific help message pointing at the lambda instead of the bare "No matching overload found".
