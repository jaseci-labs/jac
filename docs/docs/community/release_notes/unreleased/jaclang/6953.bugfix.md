- **Fix: native `zlib.decompress` of large/highly-compressible payloads (#6940 Phase 2 follow-up)**: a C `int` libz status widened to Jac's 64-bit `int` zero-extended, so `Z_BUF_ERROR` (-5) arrived as `4294967291` and `decompress`'s grow-on-demand retry never fired, so it raised on any stream whose decompressed size exceeded its input-relative initial buffer estimate. The FFI floor now folds libz status codes back to signed before comparison. Also tightens the foreign-call `bytes`→data-pointer lowering to require the declared parameter actually be an `i8*` (so a `bytes` value handed to a non-pointer slot surfaces a type mismatch instead of silently producing a pointer), and documents the libz `uLong` width (LP64 vs Windows LLP64) assumption on the FFI signatures.
