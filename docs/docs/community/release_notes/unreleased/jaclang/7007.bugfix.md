- **Bugfix: native `str.split()` / `str[i:j]` no longer leak**: On the native (`nacompile`/JIT) backend, `str.split(sep)` and string slicing `str[i:j]` allocated their results via `__rc_alloc` (refcount 1) but omitted the `_mark_owned` call the sibling single-char `str[i]` path uses, so the consuming assignment (or list-append) emitted an unbalanced extra retain and stranded every produced string at refcount 1 for the life of the process -- ~62 B per `split` element and proportional to length per slice. Both paths now mark the fresh allocation owned, and `split` releases its owned reference after the retaining list-append (mirroring the existing idiom in `lists.impl.jac`), eliminating the leak with correctness preserved.
