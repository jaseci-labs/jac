- **Refactor(jac2c): shared type-facts product with a strict build protocol**: The C (`jac2c`) and native backends now share a single `TypeFacts` product (`get_type_facts` / `get_shared_type_facts`), with `OwnershipFacts` layered on top of it, so type inference and ref-vs-value archetype classification are computed once and merged whole-program (first-wins) for cross-module virtual dispatch. The facts accessors follow one build protocol -- return the cached `module.gen` facts if present, else build them from a supplied `JacProgram`. Asking `get_type_facts` or `get_ownership_facts` for facts with no program and no cache now raises `ValueError` instead of silently returning empty facts, applying the compiler's "no fallbacks" analysis contract to the derived jac2c products. See the shared-facts layering note in the [compiler architecture doc](../../../../internals/compiler_architecture.md).
- **Refactor(jac2c): type-check the hand-written jac2c passes instead of ignoring them**: The `ownership_facts_pass`, `type_facts_pass`, and the C emitter (`c_gen_pass`) walk Jac IR, not pycparser, so they no longer belong in `.jacignore` alongside the C-ingestion frontend. They now pass the full `jac check` gate: the derived-facts accessors narrow their `any`-typed `module.gen` products with `isinstance` instead of annotated `Any` assignments, container signatures carry explicit type arguments, and `emit_translation_units`/`discover_import_graph` take a typed `JacProgram` rather than `any`. `OwnershipFactsPass.transform` no longer swallows analysis errors into empty (all-borrowed) facts -- errors now propagate to the caller's `strict_c` gate, matching `CGenPass`. Enabling the checker on `c_gen_pass` also surfaced two latent bugs, now fixed: a possible `None` dereference when reading a vtable layout, and a `.value` access on a connect/disconnect operator where only a plain binary-operator token is valid.
