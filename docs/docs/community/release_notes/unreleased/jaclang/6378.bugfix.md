- **Fix: Type checker resolves `llvmlite` instead of inferring `object`**: Added a hand-written `llvmlite.ir` type stub and a jac-owned stub search directory (`vendor/stubs`, searched ahead of importlib). The native LLVM IR-gen pass is written against `llvmlite`, which ships no type information, so the checker previously fell back to its unannotated source and reported well over a thousand false positives (`Value.type`, `PointerType.pointee`, `Type.as_pointer()`, and the `IRBuilder` builders all came through as `object`). This clears ~1400 checker errors across the native pass with no change in runtime behavior.
- **Fix: Native pass LLVM type containers typed as `object`**: The native pass tracked its LLVM struct types in `object`-typed fields (the `struct_types`/`list_types`/`dict_types`/`set_types`/`tuple_types` registries as `dict[str, object]`, and the `complex_struct_type`/`range_struct_type`/`slice_struct_type`/`jacval_type` singletons as `object`), so every lookup resolved to `object` and the `.as_pointer()` / `.set_body()` calls on them were reported as attribute errors. These fields only ever hold `ir.Type` (struct) values and are now typed precisely. Pure annotation change with no runtime effect.
