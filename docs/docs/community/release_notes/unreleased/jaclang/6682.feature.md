- **Feature: Native desktop IPC bridge for walker/function calls**: `__doWalkerFetch` and `__doFuncFetch` in the `cl` runtime now route through `globalThis.__jac_invoke` when `__JAC_NATIVE__` is set (the Jac-native desktop shell), carrying the frozen `{kind, name, node, fields, token}` request and replaying HTTP status/body semantics locally, including 401 → re-login for walkers.
- **Feature: In-process `sv` dispatch in the Jac-native desktop host** (Phase J4): `cl` → `sv` walker/function calls on the desktop target now run through the embedded CPython interpreter in-process instead of an HTTP POST to a remote backend. New `inprocess_dispatch.jac` boots the jac-scale ASGI app over a Starlette `TestClient` (no socket), registers jac-scale plugin hooks for the full `UserManager`, and serves as the native `window.__jac_invoke` trampoline with the same `{kind, name, node, fields, token}` request contract and `{status, ok, body}` reply semantics as the web API. Dispatch runs on a single serialized worker thread off the GTK UI thread so a slow walker never freezes the window, with a configurable per-call timeout (`NATIVE_DISPATCH_TIMEOUT`, non-preemptive) and a bounded queue (`NATIVE_DISPATCH_BUSY` on saturation) set via `[plugins.desktop.dispatch]` (`timeout_s`, `max_queue`) in `jac.toml`. If the in-process backend fails to boot, the host shows a deterministic native error page instead of navigating to a silently-broken app, and `dispatch()` returns a `NATIVE_BOOT_ERROR` envelope rather than hanging. Native error envelopes follow a production error contract (`{code, message, request_id}`): every error carries a `request_id` for log correlation, and full tracebacks go to stderr only -- they are echoed into the response `error.details` solely when debug is enabled via `[plugins.desktop.dispatch].debug` (or the `JAC_DESKTOP_DEBUG` env var), so release builds never leak a traceback to the client. Dev-only `JAC6596_*` runtime pinning is gated out of release builds.
