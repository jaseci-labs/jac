- **Fix: byLLM tool-call failures no longer flood logs with ERROR tracebacks**: `Tool.__call__` catches a failing tool call and hands it back to the model as a recoverable `Tool error: ...` observation, but it also logged every occurrence with `logger.exception` (ERROR level plus a full traceback, once per call, with no dedup or rate limit). A confused model repeating the same malformed tool call turned one handled, expected control-flow outcome into an unbounded ERROR-traceback flood (52 identical tracebacks in ~3 min on a single pod under load), burying genuine errors. The handled failure is now logged once at WARNING, message only, naming the tool; the observation returned to the model is unchanged.
- **Fix: truncated tool calls no longer reach the tool with their arguments silently erased**: when a response was cut off mid-tool-call, the partial arguments JSON failed to parse and `_sanitize_tool_calls` replaced it with `"{}"`, keeping no record. `"{}"` is valid JSON and indistinguishable from a genuine no-argument call, so the tool was invoked bare and the model was told it had omitted required arguments (`write_code() missing 2 required positional arguments`) rather than that its call had been truncated. byLLM already had the right observation for this ("the response was cut off ... retry with a smaller payload") but it only fired when `finish_reason == "length"`, and streaming leaves `finish_reason` empty precisely when the terminating chunk never arrives, i.e. exactly when a call was cut short. The model therefore retried the same oversized call unchanged, looping and producing the identical-failure floods described above. `_sanitize_tool_calls` now reports which calls it blanked and those are treated as truncated whatever the `finish_reason`; calls that genuinely carry no arguments are unaffected.
