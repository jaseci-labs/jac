- **Breaking: `jac start --scale` defaults `app_name`/`namespace` to the project name, not `jaseci`/`default`** (issue #7248): an unconfigured deploy no longer lands in the cluster's shared `default` namespace as app `jaseci` -- both identity values now default to a Kubernetes-safe slug of `[project].name` (e.g. project `My App` deploys as app `my-app` in namespace `my-app`), so every project gets its own namespace. Explicit `app_name`/`namespace` values still win, and projects with no usable name keep the legacy `jaseci`/`default`. **Migration for existing deploys that never set these keys** (new users and explicitly-configured projects are unaffected): the next deploy after upgrading targets the new project-named namespace with freshly provisioned empty databases, while the old deployment keeps running untouched in `default` as app `jaseci` -- including its data. Either pin the old identity in `jac.toml` to keep it (`[scale.kubernetes]` `app_name = "jaseci"`, `namespace = "default"`), or adopt the new one by tearing the old deployment down first (`jac scale destroy <entry>.jac` on the pre-upgrade version).
