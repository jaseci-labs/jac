- **Filter-comprehension value predicates push down to the storage backend**: Object-spatial filter comprehensions with value predicates (e.g. `[-->(?:User, age > 25)]`, including compound conjunctions like `[-->(?:User, age > 25, tier in ["gold", "plat"])]`) now evaluate those predicates at the storage layer instead of loading every type-matching neighbor and filtering in memory. The compiler tags each filter with its `(field, op, value)` spec (computed comparators such as `> limit()` resolve once at query time), the planner converts the pushable conjuncts into backend field predicates, and capable backends (MongoDB) run them as a single server-side query. A hybrid path keeps in-session-mutated, L1-resident nodes evaluated locally for read-your-writes correctness while pushing only cold candidates to the backend. The filter lambda is always re-applied above the results, so pushdown only ever narrows; backends without field pushdown (e.g. the SQLite dev store) are unchanged.
