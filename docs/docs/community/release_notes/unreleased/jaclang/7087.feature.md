- **Feature: bundled native `gzip` over the `zlib` floor (#6978 Phase 2)**: `import from gzip { compress, decompress }` resolves on the native (`nacompile`/JIT) pathway to a bundled `na_stdlib` module (`gzip.na.jac`) -- a pure-Jac (Mechanism B) surface that adds the RFC 1952 gzip header / CRC-32 / ISIZE framing around the raw DEFLATE body of the existing `zlib` floor, with **no new FFI**: gzip is just zlib's DEFLATE engine plus a 10-byte header and an 8-byte trailer, and the `zlib` floor's one-shot `compress2` / `uncompress` already provide the engine. `compress(data, compresslevel=9, mtime=0)` strips the 2-byte zlib header and 4-byte adler32 trailer off a zlib stream to recover the raw DEFLATE body (identical bytes under either frame) and wraps it; the output is byte-identical to CPython's `gzip.compress` at the same level and `mtime` (XFL 2/4/0 for level 9/1/other, OS byte 255). `decompress(data)` parses the gzip header (including the FEXTRA / FNAME / FCOMMENT / FHCRC flags), re-wraps the DEFLATE body as a zlib stream with a placeholder adler32, inflates it through `uncompress` -- which writes the full output and only then rejects the bogus trailer -- and re-validates with gzip's own CRC-32 + ISIZE, the canonical gzip integrity check; ISIZE gives the exact output length so a single allocation suffices. DIVERGENCES: `mtime` defaults to `0` (reproducible output) rather than CPython's wall-clock default, and `decompress` reads a single gzip member (CPython additionally concatenates trailing members); the `GzipFile` class and streaming file API are out of scope. A new equivalence fixture (`prim_gzip`, registered with `require=["na"]`) pins the native module congruent with CPython's `gzip` on compressed-stream byte-identity (poly digest + exact length, at the default and an explicit level, `mtime=0`), round-trip, and empty input.
