- **Fix: `check_pods_restarted` false-fails on stale ReplicaSet pods during redeploy (#7046)**: `check_pods_restarted` selected pods using only the `app` label, which matches pods across all ReplicaSets (current and superseded). When redeploying over a crash-looping revision, the old pod (still in `CrashLoopBackOff`) remained briefly visible while Kubernetes garbage-collected it, causing fleet-wait to report failure even though the new pod was already `1/1 Running`. The function now calls `AppsV1Api.list_namespaced_replica_set`, groups RSes by their `app` label, picks the highest-revision RS per deployment, and restricts the crash check to pods whose `pod-template-hash` matches those current RSes, making stale pods from superseded RSes invisible to the detector. If the `AppsV1` lookup fails (e.g. 403 Forbidden), the function falls back to the original broad scan rather than silently passing bad deploys. The RS hash resolution is also hoisted before the fleet-wait retry loop in `check_deployment_status`, eliminating up to 95 redundant API round-trips per fleet-wait call.
