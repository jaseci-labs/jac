- **Python-style properties are nudged toward native Jac syntax**: `@property` and the same-object `@x.getter` / `@x.setter` / `@x.deleter` now report `W2029` on every compile, suggesting the native `has x: T { getter; setter(value); ... }` form. This brings them in line with `@staticmethod` (`W2007`) and `@classmethod` (`W2006`), which were already reported this way; previously `@property` was only flagged under the opt-in `jac check --lint`. The Python-compat `class` archetype stays exempt, and `jac check --lint --fix` rewrites the decorator form automatically.
- **Fix: `[property-to-native]` left `@x.getter` behind**: the autofix folded `@property` and `@x.setter` into a native accessor block but did not recognize `@x.getter`, stranding it beside the new property as a dangling method. It is now folded in as the getter, and a cross-object `@Base.x.setter` — which extends a parent's property and has no local native form — is no longer rewritten into a getter-less property that would silently drop the parent's other accessors.
- **Fix: `[property-to-native]` corrupted impl-separated properties**: when a property's accessor bodies lived in a separate impl file, `jac check --lint --fix` rewrote the declarations into a native block but never re-targeted the impls to dotted-impl form, leaving them bound to nothing and crashing the next compile with an internal `DeclImplMatchPass` error. The impls are now re-targeted to `impl T.x.getter` / `.setter` / `.deleter`, with a getter and setter sharing the dotted target `T.x` told apart by signature.
