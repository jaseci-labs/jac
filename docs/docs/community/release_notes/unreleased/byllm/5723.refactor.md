- **Refactor: `McpClient` no longer serializes concurrent `call_tool` requests**: `_McpState.run` previously held a per-client lock across the entire `future.result()` wait, so N concurrent tool calls against the same MCP client executed strictly serially even though the underlying `mcp.client.session.ClientSession` natively multiplexes requests by JSON-RPC id. The lock now scopes only `connect()` and `close()` state transitions; `call_tool` posts its coroutine and waits on the future without holding any lock. Verified with a stdio fixture server: 5 concurrent calls each sleeping 0.5s drop from ~2.5s wall-time to ~0.5s. This is the path that makes byllm's parallel tool-calling (#5590) actually parallel for tools served by a single MCP server, which is the common shape (e.g. `jac-mcp`).
