- **Feature: distributed tracing - Tempo + Alloy collection backend (O-02.a)**. The cluster side of distributed tracing, on top of the app-side OpenTelemetry SDK. `MonitoringDeployer` gains `_deploy_tempo` (single-binary `grafana/tempo` with OTLP receivers on 4317/4318, the TraceQL API on 3200, emptyDir storage, hardened security context) as a sibling of `_deploy_loki`. The existing Grafana Alloy DaemonSet now doubles as an OpenTelemetry Collector: its River config is assembled from a header plus an optional logs lane (M-14 `loki_enabled`) and an optional `otelcol.receiver.otlp -> otelcol.processor.batch -> otelcol.exporter.otlp` traces lane (`tracing_enabled`), and a ClusterIP Alloy Service exposes the OTLP receiver so app pods have a stable target. Grafana gets a Tempo datasource (nodeGraph + tracesToLogsV2 for trace->logs by `trace_id`). Gated on a new `KubernetesConfig.tracing_enabled` flag; the microservice target sets it from `[plugins.scale.microservices.tracing].enabled`, deploys the stack for logs-only / traces-only / both, and injects `OTEL_EXPORTER_OTLP_ENDPOINT` into every pod so the app exporter reaches the in-cluster Alloy. Teardown deletes Tempo + the Alloy Service by name (idempotent). `test_tempo_collection.jac` asserts (with a mocked Kubernetes client) the `tracing_enabled` config round-trip, the Tempo ConfigMap/Deployment/Service shape with OTLP, and the `otelcol.*` lane in the Alloy config.
