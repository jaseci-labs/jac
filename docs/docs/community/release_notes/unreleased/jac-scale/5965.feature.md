- **Security + Perf: JWT token cache and credential-epoch revocation**: Authenticated requests are now faster and tokens can be revoked immediately. A process-local cache (SHA-256 keyed, 30 s TTL) eliminates repeated `jwt.decode()` calls for the same token — at 500 RPS with 100 active users this cuts cryptographic decode work by up to 80%. Token revocation is now enforced via a `tokens_valid_from` timestamp stored per user: any token issued before that boundary is rejected, covering the full token history in a single comparison. The revocation timestamp is cached per user for 60 s, so the DB is consulted at most once per user per window regardless of request rate. `invalidate_user_tokens()` is called automatically on **password change** (a compromised password can no longer extend an active session) and **account deletion** (deleted users cannot keep authenticating). New tokens include a `jti` (JWT ID) claim per RFC 7519. In multi-worker deployments, revocation propagates to other workers within the revocation cache TTL (default 60 s); reduce `_REVOCATION_CACHE_TTL` or centralise the timestamp in Redis for stricter guarantees.
