fix(jac-scale): eliminate double MongoBackend.sync() per request (issue 1g)

Every HTTP request was running two full passes over the L1 memory dict:
once inside `_finalize_call_response` (async, in thread) and again in the
jfast request-context middleware (sync, on the event loop). The second pass
found all anchors clean after the first commit updated their hashes, but
still paid O(L1-size) hash computations synchronously on the uvicorn loop.

Fix: add a `_committed` flag to `ScaleTieredMemory`. After a full async
commit completes the flag is set and subsequent `acommit()` calls return
immediately. The middleware commit is changed from `ctx.mem.commit()` to
`await ctx.mem.acommit()`, making it async and guarded by the flag.

The middleware still acts as a safety net — for error paths where
`_finalize_call_response` is never reached, and for non-walker routes
(auth, admin) that don't go through `ExecutionManager`, `_committed`
remains False and the middleware performs the commit as before.
