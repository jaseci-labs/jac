- **Feature: distributed tracing - app-side OpenTelemetry SDK (O-02.b)**. The traces leg of the observability tripod, built on K-12 trace-id propagation and M-14 structured logs. A new `jac_scale.microservices.runtime.tracing` module configures the OpenTelemetry SDK once per process (`TracerProvider` + parent-based ratio sampler + `BatchSpanProcessor` + OTLP/HTTP exporter) and opens one SERVER span per request on the gateway and each service, with the inbound W3C `traceparent` (forwarded on every proxy hop and `sv_service_call`/`sv_walker_call` RPC) as the parent so the spans nest into one trace. The K-12 `X-Trace-Id` UUID4 is reused verbatim as the 128-bit OTel trace id via a custom `IdGenerator`, so a request's spans and its `trace_id`-tagged log lines share one id (root hops produce a genuine parentless root; child hops inherit a real parent) - without this, Grafana's trace<->log pivots return nothing. OpenTelemetry is an optional dependency (`jac-scale[tracing]`, guarded through `_optdeps/opentelemetry.jac`) and the whole SDK setup is wrapped best-effort, so the gateway and services run unchanged when it's absent or disabled. Off by default; enable with `[plugins.scale.microservices.tracing] enabled = true` (optional `endpoint` / `sample_ratio`). The microservice image installs the `tracing` extra so OpenTelemetry ships in-pod. Covered by `test_tracing.jac` (id<->trace-id reconciliation, traceparent round-trip, root/child span structure, and the inactive no-op contract). The Tempo/Alloy collection backend and the in-admin Traces page land in follow-up PRs.
