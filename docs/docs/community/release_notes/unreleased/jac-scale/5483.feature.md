- **Add: Streaming sv-to-sv RPC (generator returns)**: A `def:pub` function returning an iterator now streams its yields to the caller as Server-Sent Events instead of being str-fallback-serialized. Wire format is `Content-Type: text/event-stream` with `data: {json}\n\n` framing and an explicit `event: end` terminator; producer-side exceptions are emitted as `event: error` and re-raised as `RuntimeError` out of the consumer's iterator. The consumer side (sv-RPC stub in jaclang core + jac-scale's plugin override) detects SSE by Content-Type and hands back a Python generator that yields parsed event dicts; lifecycle of the underlying httpx connection follows the generator. Retry/circuit-breaker still applies to connect failures; in-flight streams are not retried (already-consumed events cannot be replayed). Pairs with a `_finalize_call_response` fix in jaclang/runtimelib (the existing isgenerator check was on `reports`, not `result`, so explicit generator returns silently fell into the str() fallback) and a missing SSE framing wrapper in jac-scale's serve.endpoints (the StreamingResponse path emitted dict reprs instead of valid SSE).

- **Add: K5 - rolling-deploy correctness (zero-downtime deploys on K8s)**: Tightens the K2 Deployment + pod-spec with the four wires that turn "deployed" into "deployed without dropping requests." (1) `Deployment.spec.strategy = RollingUpdate{maxSurge:1, maxUnavailable:0}` ensures we surge a new pod BEFORE terminating an old one and never go below current replica count. (2) Container `readinessProbe` + `livenessProbe` both hit `/healthz` (5s period, 3s timeout, failureThreshold=3) so kube-proxy only routes to ready pods and K8s restarts hung pods. (4) `lifecycle.preStop = sleep 5` bridges the kube-proxy endpoint-propagation gap: kubelet sends SIGTERM and updates the Endpoints object simultaneously, but other nodes' kube-proxy tables don't reflect the removal for a few seconds; without the sleep, the pod stops accepting work BEFORE remote nodes stop sending it work, producing 503s for clients on existing connections. Together with the existing drain machinery from P13, rolling deploys complete with zero non-2xx during the rollout window. - **Fix: gateway `/healthz` no longer falls into builtin passthrough**: `/healthz` was in the gateway dispatcher's `_BUILTIN_EXACT` set, so the K5 readinessProbe path tried to fan out to a backend service - which meant a freshly-started gateway pod (before any backend had registered) returned 404 to kubelet and never went Ready. K8s convention is that `/healthz` is the pod's own liveness, not a fan-out probe; this aligns the gateway with that contract.
- **Fix: `start_gateway_only` pre-marks registry entries HEALTHY in K8s mode**: `build_registry` initialises every entry to `ServiceStatus.REGISTERED`, and in local mode `LocalDeployer.deploy_service` flips it HEALTHY after a successful spawn. - **K5 v2: split `/healthz/ready` vs `/healthz/live` so liveness doesn't trip on dependency degradation**: K5 v1 used a single `/healthz` for both probes, which is production-dangerous - a transient downstream blip (DB hiccup, peer service restart) trips the liveness probe, kubelet restarts the pod, the restart blips other pods, and you get a cascade. v2 splits into `/healthz/live` (liveness, gates pod restart, deliberately checks NOTHING other than dispatcher responsiveness) and `/healthz/ready` (readiness, gates Endpoints membership, returns 503 during graceful drain so kube-proxy stops routing). - **Test: rolling-deploy zero-downtime verification across two phases (gateway + service)**: K5 has claimed "zero requests dropped during rolling restart" since v1, with a single hammer-then-restart-the-gateway assertion at the end of the real-app e2e.
