- **Add: Streaming sv-to-sv RPC (generator returns)**: A `def:pub` function returning an iterator now streams its yields to the caller as Server-Sent Events instead of being str-fallback-serialized. Wire format is `Content-Type: text/event-stream` with `data: {json}\n\n` framing and an explicit `event: end` terminator; producer-side exceptions are emitted as `event: error` and re-raised as `RuntimeError` out of the consumer's iterator. The consumer side (sv-RPC stub in jaclang core + jac-scale's plugin override) detects SSE by Content-Type and hands back a Python generator that yields parsed event dicts; lifecycle of the underlying httpx connection follows the generator. Retry/circuit-breaker still applies to connect failures; in-flight streams are not retried (already-consumed events cannot be replayed). Pairs with a `_finalize_call_response` fix in jaclang/runtimelib (the existing isgenerator check was on `reports`, not `result`, so explicit generator returns silently fell into the str() fallback) and a missing SSE framing wrapper in jac-scale's serve.endpoints (the StreamingResponse path emitted dict reprs instead of valid SSE).
- **Add: Configurable gateway-to-service forward timeout**: `[plugins.scale.microservices].http_forward_timeout` (float seconds, default 30) controls the aiohttp timeout used by `raw_forward` (built-in passthrough fan-out) and `stream_forward` (path-routed proxy). Per-service overrides at `[plugins.scale.microservices.services.NAME].http_forward_timeout` mirror the existing `rpc_timeout` precedence pattern - useful for LLM/long-running services that need minutes rather than the global default. Distinct from `rpc_timeout`, which still controls inter-service `sv import` calls (httpx); these are two different code paths through two different HTTP clients. `jac setup microservice` emits a commented reference block.
- **Add: K1 - Kubernetes microservice deployment scaffold**: New `KubernetesMicroserviceTarget(KubernetesTarget)` reuses the existing K8s image-build/push pipeline and iterates over `[plugins.scale.microservices.routes]` to emit per-service pod-spec stubs. Single image is shared across all services and the gateway; pods are differentiated at runtime by `JAC_SV_NAME` env var. Wired into the existing `jac start <file> --scale` flow: when `[plugins.scale.microservices].enabled = true` and the user accepts the default target, `_scale_pre_hook` automatically swaps `target=kubernetes` to `target=kubernetes-microservice`. Registered as a built-in target in `DeploymentTargetFactory`. K1 deliberately stops at "build image + emit pod-spec dicts" - returns them in `DeploymentResult.details["pod_specs"]` for inspection. Manifest generation (K2), kubectl apply (K3), runtime DNS adapter (K4), and rolling-deploy correctness (K5) follow.
- **Add: K2 - per-service Deployment + ClusterIP Service manifest generation**: `KubernetesMicroserviceTarget.generate_manifests(app_config)` lifts K1's pod-spec stubs into full K8s manifest dicts. Each service (and the gateway) gets a paired `apps/v1 Deployment` (replicas=1 placeholder; K6 will make per-service-configurable) wrapping the K1 pod-spec, and a `v1 ClusterIP Service` whose selector matches the Deployment's pod labels so kube-proxy routes inter-service traffic correctly. New `_k8s_safe_name` helper maps the `__gateway__` sentinel to a DNS-1123-valid `gateway` resource name and hyphenates underscores in user-defined service names. Manifests carry `app=<name>`, `managed=jac-scale`, and `jac-scale.role=microservice|gateway` labels matching the existing `KubernetesTarget` convention. K2 still does NOT call kubectl - it returns the manifest bundle in `DeploymentResult.details` with `deployments`, `services`, and `pod_specs` sections. K3 will iterate the bundle and apply via kubectl in dependency order.
- **Add: K3 - apply/destroy lifecycle via the kubernetes Python client**: `KubernetesMicroserviceTarget.deploy()` now actually applies the K2 bundle. `apply_manifests(bundle)` ensures the namespace exists, then iterates Services first (so DNS resolution works the moment a Deployment's first pod becomes ready) and Deployments second. Each apply is idempotent: read-then-patch on hit, create on 404. `destroy(app_name)` tears down the topology via a label selector (`managed=jac-scale,jac-scale.role in (microservice,gateway)`) instead of per-name deletion, so renamed services in `jac.toml` still get cleaned up. Kube config is loaded from kubeconfig outside the cluster and from in-cluster service-account config inside a pod, matching the existing `KubernetesTarget` pattern. K5 will tighten the Deployment manifest with `RollingUpdate { maxSurge, maxUnavailable }`, readinessProbe gating, `terminationGracePeriodSeconds` aligned with our `drain_timeout_seconds`, and a preStop sleep to bridge the kube-proxy endpoint-propagation gap (the wires that turn "deployed" into "deployed without dropping requests").
- **Add: K4 - runtime DNS adapter (`KubernetesDeployer` + `get_sv_registry` hookimpl)**: New `KubernetesDeployer(ServiceDeployer)` mirrors `LocalDeployer`'s surface but `url_for(svc)` returns `http://{svc}-service.{namespace}.svc.cluster.local:{port}` instead of a localhost loopback. Selected at runtime by a new `get_sv_registry` `@hookimpl` in `JacScalePlugin` that detects K8s-in-cluster via the `KUBERNETES_SERVICE_HOST` env var (which K8s injects into every pod automatically). When in cluster, builds a DNS map from the configured routes table; namespace comes from `POD_NAMESPACE` env (downward API) with a `/var/run/secrets/.../namespace` file fallback then `default`. When NOT in cluster, falls back to core's default behavior (local `sv_client._registry`). The gateway needs zero code changes - `gateway.resolve_target_url` already calls `JacRuntime.get_sv_registry()` and gets the right URLs based on environment. Lifecycle methods (`deploy_service`, `stop_service`, `restart_service`, `scale_service`) are stubbed in K4 because pod lifecycle is owned by `KubernetesMicroserviceTarget` at deploy time and by the K8s controller at runtime; in-cluster admin commands (`jac scale stop orders` from inside an admin pod) are a future K-task. Also extracted `_k8s_safe_name` from `KubernetesMicroserviceTarget` to module-level `k8s_safe_name` in `k8s_target.jac` so both target + deployer can share the DNS-1123 normalization without duplication.
- **Add: K5 - rolling-deploy correctness (zero-downtime deploys on K8s)**: Tightens the K2 Deployment + pod-spec with the four wires that turn "deployed" into "deployed without dropping requests." (1) `Deployment.spec.strategy = RollingUpdate{maxSurge:1, maxUnavailable:0}` ensures we surge a new pod BEFORE terminating an old one and never go below current replica count. (2) Container `readinessProbe` + `livenessProbe` both hit `/healthz` (5s period, 3s timeout, failureThreshold=3) so kube-proxy only routes to ready pods and K8s restarts hung pods. (3) `terminationGracePeriodSeconds = drain_timeout_seconds + 5` (sourced from `[plugins.scale.microservices].drain_timeout_seconds`) gives our P13 drain middleware enough time to flush in-flight requests plus a 5s cushion for uvicorn to exit AFTER drain returns. (4) `lifecycle.preStop = sleep 5` bridges the kube-proxy endpoint-propagation gap: kubelet sends SIGTERM and updates the Endpoints object simultaneously, but other nodes' kube-proxy tables don't reflect the removal for a few seconds; without the sleep, the pod stops accepting work BEFORE remote nodes stop sending it work, producing 503s for clients on existing connections. Together with the existing drain machinery from P13, rolling deploys complete with zero non-2xx during the rollout window. K5 v1 uses `/healthz` for both probes; a future task should split into `/healthz/ready` (gates traffic) and `/healthz/live` (gates restart) so liveness doesn't trip on dependency degradation. K6 will add per-service HPA + PodDisruptionBudget on top of these manifests.
- **Add: K6 - HorizontalPodAutoscaler + PodDisruptionBudget per service**: Each service Deployment (and the gateway) gets its own `autoscaling/v2 HorizontalPodAutoscaler` (default min=1, max=3, CPU avg utilization 70%) and `policy/v1 PodDisruptionBudget` (default `maxUnavailable: 1`). HPA's `scaleTargetRef` points at the matching `<name>-deployment`, so the K8s autoscaling controller scales the right Deployment up under CPU pressure and back down when idle. PDB uses `app=<name>` as the pod selector, matching the Deployment's pod-template labels. Together with K5's `RollingUpdate{maxSurge:1, maxUnavailable:0}`, voluntary disruptions (rollouts, node drains, evictions) stay bounded: the rolling-update settings handle scheduled disruptions, the PDB handles unscheduled ones. `apply_manifests` now applies HPAs + PDBs after Deployments (since both reference Deployment names / pod labels); `destroy` cleans them up via `delete_collection` on the same `managed=jac-scale` label selector. K7 will let users tune or disable HPA/PDB per service via `[plugins.scale.microservices.services.NAME.{hpa,pdb}]`.
- **Add: K7 - per-service config layering for K8s manifests**: `[plugins.scale.microservices.services.NAME]` now controls the per-service shape of the generated K8s manifests. New keys: `replicas` (Deployment.spec.replicas, default 1); `cpu_request` / `cpu_limit` / `memory_request` / `memory_limit` (container resources, partial subsets emit only what's set); `env` (extra container env vars, merged with the auto-set JAC_SV_NAME which is protected from override); `image_tag` (overrides the global tag for canary, correctly handling registry hostnames with ports like `myregistry.io:5000/name`); nested `hpa.{enabled,min,max,cpu_target}` and `pdb.{enabled,max_unavailable}` sub-tables. `enabled = false` on either skips emitting that manifest entirely. The gateway uses the same table keyed by `__gateway__`. Defaults match K5/K6 exactly so omitting all keys gives the same manifests as before. New helpers in `KubernetesMicroserviceTarget`: `_service_config(svc_name)` reads the per-service section, `_build_env`, `_build_resources`, `_resolve_image` apply the layered overrides. Schema description in `plugin_config.jac` updated with all the new keys. K8 (minikube e2e + CI) is the last K-task.

- **Fix: gateway `/healthz` no longer falls into builtin passthrough**: `/healthz` was in the gateway dispatcher's `_BUILTIN_EXACT` set, so the K5 readinessProbe path tried to fan out to a backend service - which meant a freshly-started gateway pod (before any backend had registered) returned 404 to kubelet and never went Ready. The dispatcher now handles `/healthz` directly as an alias for `/health`, returning the gateway's own registry summary with status 200 regardless of backend state. K8s convention is that `/healthz` is the pod's own liveness, not a fan-out probe; this aligns the gateway with that contract.
- **Fix: `start_gateway_only` pre-marks registry entries HEALTHY in K8s mode**: `build_registry` initialises every entry to `ServiceStatus.REGISTERED`, and in local mode `LocalDeployer.deploy_service` flips it HEALTHY after a successful spawn. K9's `start_gateway_only` skips the orchestrator entirely (K8s owns service lifecycle), so entries stayed REGISTERED forever and `handle_proxy` short-circuited every `/api/<svc>/*` request with 503. Mark entries HEALTHY in `start_gateway_only` so route matches forward through to `stream_forward`; transport-layer errors from a not-yet-Ready pod still bubble up naturally (kube-proxy only routes to Ready pods anyway, so the gateway's internal status field is the wrong layer to enforce health in K8s).
- **Fix: `get_microservices_config` returns the `ingress` block**: K10's `_build_ingress_manifest` reads the ingress section through `get_scale_config().get_microservices_config()`, but the impl was building an explicit return dict and never including the `ingress` key. Result: even when users set `ingress.enabled = true` in jac.toml, the K8s target saw `{}`, returned None from the builder, and the Ingress object was silently never applied. Pass the ingress sub-table through opaquely so the K10 builder sees the user's host / ingress_class_name / annotations.
